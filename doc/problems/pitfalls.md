## 二、特征工程阶段

| # | 坑 | 影响 | 对策 |
|---|-----|------|------|
| 9 | **大盘特征 look-ahead bias**。如果当天 9:31 的个股分钟线注入了当天指数的**全天**走势数据，属于未来信息 | 模型训练时看到了"未来"，推理时拿不到同样信息，泛化严重失真 | 大盘特征只注入**已发生**的指数分钟数据。即 9:31 个股只对应 9:31 指数的数据。推理时同样从实时行情获取 |
| 10 | **板块分类数据源不稳定**。东方财富的板块分类经常调整，同一只股票 3 个月前属于"消费电子"，现在属于"AI智能" | 板块特征训练时和推理时不一致 | 使用固定的申万一级行业分类（每年调整一次，可追溯）。或使用 zzshare 提供的板块API |
| 11 | **个股与指数时间对齐**。个股 9:30 第一根K线，指数也是 9:30，但 zzshare 返回的时间戳格式可能一个是 `20250601 09:30` 另一个是 `2025-06-01T09:30:00` | merge 失败或错误对齐 | 统一用 `pd.Timestamp` 并 strip 到分钟精度。特征工程入口处统一格式化 |
| 12 | **情绪特征的单日广播**。同一交易日所有 240 根分钟线的 sentiment 值相同（广播注入），LSTM 可能学不到情绪变化的时序模式 | 情绪特征在一天内是常量，提供的信息量等于 1/240 | 改为注入情绪K线的 OHLC 四维（而非单一 close）。或计算前 N 日情绪的变化率作为额外特征 |
| 13 | **特征维度冗余**。26 维个股 + 5 维大盘/板块 = 31 维。其中 ma5/ma10/ma20 高度相关（共线性 > 0.9） | 模型过拟合，泛化差。而且 31 维对 240 步长的 LSTM 来说参数量大 | 训练前做特征重要性分析（随机森林 feature_importance）。必要时去掉高度相关特征或做 PCA 降维 |
| 14 | **增量计算时前向填充的边界**。新增原始K线的前几行需要历史数据计算 MA/RSI/MACD | 新数据的前几行指标值为 0（被 fillna(0)），与历史数据的指标值不连续 | 增量计算时向前多取 120 条原始K线做 warm-up，计算完指标后只保留新增部分写入 |
| 15 | **volume 字段类型不一致**。zzshare 返回的 volume 可能是 int、float 或 str | 类型转换失败卷 NaN | 统一 `pd.to_numeric(..., errors='coerce').fillna(0).astype(int)` 处理 |

## 三、回测引擎阶段（最复杂，坑最多）

### 3.1 多仓位资金池模型

**最终确定的模型**：
```
单资金池 50,000
├── 现金: 动态变化
├── 持仓: 多个批次（不同日期、不同价格、不同数量）
│   ├── 批次1: D1买入 10手 @10.50 (entry_date=D1, T+1→D2可卖)
│   ├── 批次2: D2买入 5手 @10.80 (entry_date=D2, T+1→D3可卖)
│   └── 批次3: D3买入 8手 @10.30 (entry_date=D3, T+1→D4可卖)
└── 窗口末: 总资产 = 现金 + 各批次市值
```

**规则**：
- 买入比例：策略参数之一（Optuna搜索），占可用现金的百分比上限
- 卖出比例：策略参数之一（Optuna搜索），占当前持股的百分比上限
- 信号优先级同分钟：风控信号(止盈止损) > 卖出信号 > 买入信号
- T+1 按批次独立计算，卖出时 FIFO（优先卖最早的批次）
- 窗口末不强制平仓，未平仓按市价计入总资产

| # | 坑 | 影响 | 对策 |
|---|-----|------|------|
| 16 | **批次管理复杂度**。每个批次需要记录：`{entry_price, entry_bar, entry_date_idx, shares, highest_close}`。5天窗口可能累积 10-20 个批次 | 内存和计算开销增大 | 批次用 namedtuple 或 dataclass，卖出时遍历批次列表查找可卖的（`entry_date_idx < cur_date_idx` 且 shares > 0）。列表长度可控 |
| 17 | **FIFO 卖出时的拆分**。卖出 8 手，但最早批次只有 5 手。需要拆成 5+3 卖两个批次 | 需要实现部分卖出逻辑 | 遍历批次直到满足卖出数量，对被打到的最后一个批次做部分扣除而非全删 |
| 18 | **买入时资金不足**。触发买入信号但现金不够买 1 手（100股）| 信号浪费，回测低估策略机会 | 买入前检查：`buy_amount = cash * buy_ratio`，`shares = floor(buy_amount / (price * 100)) * 100`，shares > 0 才执行 |
| 19 | **印花税只在卖出时收取**。当前代码买入卖出各扣一次 commission。实际 A 股：买入收佣金，卖出收佣金 + 印花税（千一）| 卖出成本被低估约 0.05% | 买入手续费 = 佣金(0.001) + 滑点(0.0005)。卖出手续费 = 佣金(0.001) + 滑点(0.0005) + 印花税(0.001) |
| 20 | **收益计算方式改变**。多批次 + 不强制平仓 → 不能再按"交易序列"算 Calmar | 需要用资金曲线方法 | 每根K线计算：`总资产 = 现金 + sum(各批次 shares × 当前 close)`。形成一条 2400 根K线的净值曲线，从曲线算 max_dd 和总收益 |
| 21 | **ATR(14) 在分钟线上值太小**。分钟线 ATR(14) 可能 0.01-0.1 元，stop_atr × ATR 止损线太近，正常波动就触发止损 | 几乎所有交易都以止损结束，胜率极低 | ATR 周期改为 240（日级别），或直接用日线 ATR 按比例映射到分钟线。`atr_1m = atr_daily / sqrt(240)` |
| 22 | **各策略信号在分钟线上过于频繁**。MA 金叉可能一天触发 5 次假突破 | 噪音开仓过多，摩擦成本吃掉利润 | 加入冷却期：同一策略方向，上次开仓后至少间隔 N 分钟才允许再次开仓。冷却期也是 searchable parameter |
| 23 | **涨跌停检测的边界情况**：
> - 一字板涨停：open=close=high=low=涨停价，全天无买入机会
> - 地天板：从跌停到涨停日内波动 20%+
> - 连续涨停：次日开盘即涨停，T+1 下前日买入的批次无法卖出 | 回测高估收益（以为能在涨停价买到/卖到） | 一字板检测：`open==high==low==close` 且 `all >= limit_up*0.99` → 该分钟不可买入。连续涨停时，持仓批次无法卖出的场景正确模拟（跌停同理） |
| 24 | **has_gap 跨日误判**。周五 15:00 → 下周一 9:30 间隔 66 小时，也可能因为假期更长。半日市 11:30 收盘 → 次日 9:30 间隔 >22 小时 | 正常交易日被误判为停牌跳过 | gap 检测改为**仅检测同日内的断层**。跨日 gap 用交易日列表判断：如果两个连续日期都在交易日列表中，则是正常跨日 |
| 25 | **不同板块的涨跌停比例**：
> - 主板（sh60/sz00）：±10%
> - 创业板（sz30）：±20%
> - 科创板（sh688）：±20%
> - 北交所（bj）：±30%
> - 股票简称含 "ST"：±5% | 硬编码 10% 导致创业板回测误判 | 根据 `stock_code` 前缀判断板块，返回对应的涨跌停百分比。ST 检测通过股票名称判断 |
| 26 | **滑点建模过于简单**。当前用固定百分比，但实际滑点取决于流动性和波动率 | 小盘股和高波动时的滑点被低估 | 滑点 = 基础滑点 + f(成交量, 波动率)。简化为：如果当前 `volume < volume_ma20 * 0.5`（流动性差），滑点 × 2 |
| 27 | **回测窗口首根K线的前收盘价**。`pre_close = close[0]` 用于计算涨跌停线，但如果窗口第一分钟是 9:30，pre_close 应该是昨日收盘价 | 涨跌停线计算偏移 | 回测窗口向前多取 1 根K线作为 pre_close。如果取不到（数据开头），用 `open[0]` 近似 |

| # | 坑 | 影响 | 对策 |
|---|-----|------|------|
| 28 | **线程池中 Optuna study 对象不线程安全**。多个线程共享同一个 study 会导致数据竞争 | 结果混乱，Calmar 排序错误 | 每个线程创建独立 `optuna.create_study()` + 独立回测引擎实例。结果汇总到线程安全的 queue |

| 30 | **Calmar 正样本阈值选择**。固定阈值（如 >0.5）在不同市场环境下含义不同。牛市 0.5 太容易，熊市 0.5 几乎不可能 | 正样本分布随市场环境剧烈漂移 | 动态阈值：每个回测窗口结束后，用该窗口所有 trial 的 Calmar 分位数 p75 作为正样本的判定尺度。或使用 rank-based 标签（而非 absolute value-based） |
| 31 | **179 只股的样本时间跨度不同**。有些股票最近 100 天才有数据 | 全局时间 80/20 切分时，某些股票全被分到训练集（或验证集），导致 out-of-distribution | 按每只股票内部的时间排序 → 各切 80/20 → 再全局 concat。保证每只股票都有训练和验证样本 |
| 32 | **样本生成中途崩溃**。179 只股票 × 200 窗口 × 5 策略 = 大量计算。任何异常（TDengine断连、Optuna报错、磁盘满）都会中断 | 跑了几个小时后白跑 | 每只股票完成后写 `.parquet` 文件作为断点标记（v3 已实现）。增加 try-except per window（单窗口失败不影响整只股票） |
| 33 | **TDengine 连接池**。线程池 5 线程同时查询 TDengine，单个连接并发查询可能阻塞 | 线程池效率打折扣 | 连接池大小 = max_workers + 1（5+1=6）。`queue.Queue` 管理，线程从池中取出用完归还。连接池在脚本入口初始化一次 |
| 34 | **日志量爆炸**。179 只 × 200 窗口 = 35,800 窗口。如果每个窗口都打一条 INFO | 日志文件 100MB+，磁盘满 | 分级日志：每 50 窗口打一次汇总（平均 Calmar、正样本数）。每条样本打 DEBUG（默认关闭）。`--verbose` 开启 DEBUG |

## 五、模型训练阶段（预先提示）

| # | 坑 | 影响 | 对策 |
|---|-----|------|------|
| 35 | **双向 LSTM 泄露未来信息**。推理时没有"未来"的K线数据，但训练时双向 LSTM 看到了 | 训练 Loss 低但推理准确率灾难 | **v2.2 改为单向 LSTM**。或需要在推理时也传入未来数据（不可能） |
| 36 | **240 步长 LSTM 梯度消失**。早期时间步（如开盘 9:30 的特征）通过 240 个时间步传到最后一个 hidden state 时梯度几乎为 0 | 模型学不到开盘时的信息，只依赖尾盘 | 加 skip connection（输入特征直接传到全连接层）或改用 Transformer（self-attention 没有长距离衰减） |
| 37 | **Calmar 值域极宽**（-100 到 1000）。作为 loss 中的 weight 项会导致 loss 被几个极端样本主导 | 训练不稳定，梯度爆炸 | Calmar 做 clip(-10, 10) + log(1+abs(calmar)) * sign(calmar) 变换，压缩值域 |
| 38 | **5 个策略的参数 p1/p2 含义完全不同**。MA 突破的 p1=快线周期（2-20），放量突破的 p1=量比（1.5-5.0）。同一个 Linear(64→5) 头预测含义不同的值 | 参数回归 Loss 无法下降，多头学习冲突 | v2.2 暂不改：模型先只做策略分类（CrossEntropy），参数由 Optuna 搜索到的中位数或规则确定。后续加 5 个独立的参数回归头 |
| 39 | **features_blob 存储格式**。当前用 `np.save` 序列化为 bytes 存入 parquet。每条样本 ~25KB，70 万条 ≈ 17.5GB | 训练时读取极慢（解压 numpy + 重组为 tensor） | 改为 hdf5 或 zarr 数组存储特征矩阵，parquet 只存元数据 + 特征索引指针。训练时直接 mmap 读取 |

## 七、多仓位回测引擎最终模型

```
┌───────────────────────────────────────────────────┐
│              单资金池: 50,000 元                    │
│  ┌──────────┐  ┌──────────────────────────────┐   │
│  │   现金    │  │     持仓批次 (FIFO)            │   │
│  │ 动态变化  │  │                               │   │
│  │          │  │  批次1: D1 10手@10.50, T+1→D2 │   │
│  │   ───────│  │  批次2: D2 5手@10.80,  T+1→D3 │   │
│  │  买入←───│──│  批次3: D3 8手@10.30,  T+1→D4 │   │
│  │  ───────→│  │  批次4: D4 3手@11.20,  T+1→D5 │   │
│  │  卖出     │  │                               │   │
│  └──────────┘  └──────────────────────────────┘   │
│                                                    │
│  信号优先级 (同分钟):                               │
│    止盈/止损/超时 (风控) > 卖出信号 > 买入信号       │
│                                                    │
│  T+1: 按批次独立, 买入日次日才能卖出该批次          │
│  卖出: FIFO (优先卖最早批次), 部分卖出拆分批次      │
│                                                    │
│  买入参数: buy_ratio (Optuna搜索, 占可用现金%)      │
│  卖出参数: sell_ratio (Optuna搜索, 占当前持股%)     │
│  冷却期:  N分钟 (Optuna搜索, 同策略间隔)            │
│                                                    │
│  窗口末: 总资产 = 现金 + Σ(批次股份 × 收盘价)       │
│  绩效:   从每日净值曲线计算 max_dd → Calmar         │
└───────────────────────────────────────────────────┘
```
